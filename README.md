# docs-as-code Workflow Pack

This repository contains a reusable workflow pack for turning an empty folder plus one product document into a governed, docs-as-code project workspace.

Current release identity comes from `VERSION`; reviewed consumer-facing changes and upgrade impact are recorded in `CHANGELOG.md`.

## Goal

Create reliable project governance before and during implementation:

- archive the original product document
- record product archive manifest metadata and SHA-256 evidence
- copy a target-local workflow, skill, reference, and template snapshot
- derive structured product, architecture, API, UI, backend, frontend, test, and delivery documents
- keep unresolved decisions explicit
- verify documentation structure and drift
- hand implementation tasks to agents with traceable specs and acceptance criteria
- execute Ready tasks with scoped code changes and local verification evidence

## Package Layout

```text
.
├── CHANGELOG.md # reviewed release changes and upgrade impact
├── bin/          # command wrappers
├── scripts/      # deterministic checks and bootstrap utilities
├── skills/       # agent skills used by the workflow
├── references/   # supporting methods and practice references
├── templates/    # generated repository document templates
├── tests/        # workflow-pack tests

```

## Quick Start

Before using the pack on a real project, run the disposable source-pack dry run. It creates a temporary target, imports a sample product document, executes the generated target-local Make command surface, records reviewed PRD-hash-bound dispositions for unsupported optional product chapters, proves the product work package advances to the phase action, advances through product structuring and design derivation, builds the design authoring queues, records complete source/evidence/authority-bound `design_reviews`, and executes current API, threat, reliability, and migration review gates. It then advances a minimal source-backed target into implementation and proves the guarded runner claims one task and stops for edits, preflights every bound command before execution, runs all required checks, applies snapshot-guarded closeout, and preserves completion through runtime refresh:

The implementation gate remains blocked until placeholders and authority reviews are complete; closeout blocks `Done` until passing local evidence exists for every required command.

```bash
make dry-run
make dry-run-golden
python3 scripts/dry_run_workflow.py --json
python3 scripts/dry_run_workflow.py --product tests/fixtures/product-docs/field-service-ops.md --json
```

Use the explicit product fixture command when you want a multi-acceptance dry run that proves API/backend/frontend/test/planning/ADR queues scale with more than one product acceptance criterion.

Run the compact real-stack gate when release or CI evidence needs an implementation acceptance matrix instead of the full dry-run payload:

```bash
make stack-acceptance
python3 scripts/stack_acceptance.py --json
python3 scripts/stack_acceptance.py --strict-rust --json
```

The default gate runs dependency-free Python and Node tests through the reviewed project-environment and implementation-evidence chain. Rust runs offline Cargo tests when available and otherwise reports reviewed manual repair routing; `--strict-rust` makes that Rust result mandatory. These fixtures use no network access or third-party packages.

To hand this workflow pack to another workspace or agent environment, export a manifest-checked source pack:

```bash
make package
python3 scripts/export_workflow_pack.py --check --json
python3 scripts/export_workflow_pack.py --output dist/docs-as-code-workflow-pack --archive dist/docs-as-code-workflow-pack.tar.gz --force --json
python3 scripts/verify_pack_manifest.py dist/docs-as-code-workflow-pack --json
```

The export writes `pack-manifest.json` with SHA-256 evidence for every included source-pack file, runs `verify_pack` against the exported directory, and creates a tar.gz artifact that can be unpacked as a trusted source workflow-pack checkout.
The manifest verifier validates `pack-manifest.json` by recomputing SHA-256 hashes, `size_bytes`, executable flags, duplicate or invalid paths, and unmanifested files.
Run `make ci` locally as the authoritative source-pack baseline. The source workflow pack also carries a manual-only `.github/workflows/ci.yml`, which pins Python 3.10 and Node 22, then runs `make test`, `make stack-acceptance`, `python3 scripts/verify_pack.py --json`, and `python3 scripts/check_env.py --json` only after an explicit `workflow_dispatch`. It has no `push` or `pull_request` trigger, so normal repository activity does not consume GitHub Actions; use it only when a reviewed remote-environment run is specifically required.
Use `python3 scripts/authority_skills.py --json` or `make authority-skills` for an offline inventory of authority-routing specialist skills. Run `python3 scripts/authority_skills.py --repair --check --json` to classify `current`, `missing`, `drifted`, `unmanaged`, and `source-unregistered` skills and build a no-write repair plan from `references/authority-skills.lock.json`. The checked-in lock registers every required skill from reviewed community sources at immutable commits and binds each complete skill tree to a SHA-256 digest; it never guesses a repository from a skill name. After reviewing the plan, run `python3 scripts/authority_skills.py --repair --apply --approve-installs --strict-provenance --json` to approve all eligible missing-skill installs once. Apply mode refuses drifted, duplicated, unmanaged, source-unregistered, or unavailable-installer actions, executes without a shell, and verifies each installed tree before continuing. Use `--strict` for availability and `--strict-provenance` when every required skill must be installed and current.

After a target has been initialized, use `bin/governance repository init <target> --default-branch <branch> --author-name "<name>" --author-email "<email>" --reviewed --check --json` and then its write-mode equivalent to create local Git metadata. Add `--origin <url>` only when reviewed. The command sets repository-local author identity, blocks parent repositories and conflicting metadata, and never commits, authenticates, or pushes; verify the credential's hosting account separately before the first push.

To prove the transfer artifact is self-contained, smoke-test the tar.gz by unpacking it and running checks from the unpacked workflow pack:

```bash
make artifact-smoke
python3 scripts/smoke_workflow_pack_artifact.py --json
python3 scripts/smoke_workflow_pack_artifact.py --archive dist/docs-as-code-workflow-pack.tar.gz --json
```

The default smoke command exports a temporary archive first, then unpacks the tar.gz artifact for validation. Pass `--archive dist/docs-as-code-workflow-pack.tar.gz` to validate the exact release artifact that will be handed off. The smoke test requires `consumer_bootstrap_one_command.ok: true` and `consumer_bootstrap_one_command.repository_git_initialized: true` after running the unpacked `bin/governance-bootstrap` check/apply path against a product-only target; it also proves the configured branch and local author while confirming no commit was created. It requires `consumer_bootstrap_product_conversion.ok: true` after converting an unpacked TXT product source into hash-bound review evidence. It initializes another fresh target folder, runs target-local verify/status/workflow-plan/work-package/workflow-resume commands, exercises consumer bootstrap with `--auto-repair-env --workflow-preset product-structure`, `--auto-repair-env --workflow-preset design-scaffold`, `--auto-repair-env --workflow-preset design-routing`, and `--auto-repair-env --workflow-preset implementation-routing`, and then runs the full dry-run workflow from the unpacked workflow pack. The exported-pack dry run must invoke consumer `--resume` after design evidence is complete and report `consumer_resume_implementation_handoff.ok: true`. Also require `implementation_run.complete: true`, `stack_acceptance.ok: true`, `design_reviews.ok: true`, `api_review.ok: true`, `threat_review.ok: true`, `reliability_review.ok: true`, and `migration_review.ok: true`.

Before tagging or handing off a workflow-pack release, run the release readiness gate from `references/release-readiness-checklist.md`:

```bash
make release-check
python3 scripts/release_readiness.py --json
```

The release readiness payload includes a non-strict `authority-skill-inventory` criterion so the required specialist-skill routing remains auditable without making clean CI depend on locally installed agent skills.

For a recipient environment that has unpacked the source workflow-pack artifact, compose the source-pack checks, environment preflight, target initialization, and target-local verification with the consumer bootstrap script:

For the shortest safe startup, place exactly one supported product document and the unpacked `docs-as-code-workflow-pack/` directory in the new project folder, change to that folder, then run:

```bash
./docs-as-code-workflow-pack/bin/governance-bootstrap --check --json
./docs-as-code-workflow-pack/bin/governance-bootstrap --json
```

The wrapper preserves the project folder as the working directory and enables the existing safe `--auto-repair-env` policy; check mode remains no-write, while write mode applies only repairs that require neither approval nor manual action. With no `--target`, consumer bootstrap selects the current directory; with no `--project-name`, it records the target directory name; with no `--product`, it uses target-root auto-discovery and proceeds only when exactly one supported product document exists. It does not infer a stack profile, so the default remains `unknown` unless `--profile` is supplied. JSON `input_resolution` records every explicit or defaulted input. The workflow-pack root and its descendants are never accepted as consumer targets; placing the pack inside the target, as shown above, remains valid. Use the longer form below when paths, names, profile, authority approval, environment repair, or workflow preset must be explicit.

Before any pack or target command, the wrapper requires Python 3.10 or newer. Set `DOCS_AS_CODE_PYTHON` to an already installed compatible interpreter when `python3` is not the right executable. In JSON mode, an absent interpreter returns `bootstrap_python_unavailable`; a broken or older interpreter returns `bootstrap_python_incompatible`. Both responses set `writes_state: false` and return a `manual-runtime-repair` route because the workflow cannot safely install the runtime required to inspect the remaining environment. When the variable remains set, generated target-local `bin/governance`, its `governance-init` and `governance-verify` aliases, environment inventory, and the `core-governance` version probe use the same interpreter. Target-local failures return the equivalent no-write `governance_python_unavailable` or `governance_python_incompatible` code. All `bin/` wrappers use POSIX `/bin/sh`; Bash is not required. After a successful probe, the wrapper uses `exec` so the Python process owns signals and the final exit code.

To create the local Git repository in the same reviewed bootstrap, use the existing project folder as the target and run the check/apply pair with explicit repository-local identity:

```bash

./docs-as-code-workflow-pack/bin/governance-bootstrap --initialize-git --git-default-branch main --git-author-name "<name>" --git-author-email "<email>" --git-origin "<url>" --reviewed-git --json
```

Omit `--git-origin` when no remote has been reviewed. `--initialize-git` requires the target directory to exist and requires explicit branch, author name, author email, and `--reviewed-git`; Git metadata flags are rejected without it. Check mode must report `repository_git_check_ok: true` without creating `.git`. Apply runs only after governance initialization, reports `repository_git_initialized: true`, and creates no commit, performs no remote authentication, and makes no push attempt. Verify the credential helper's hosting account separately before the first push.

TXT sources are converted automatically into reviewable Markdown with `docs/product/core/source/conversion-report.json` hash evidence. DOCX and HTML first elevate only Pandoc through `--require-tool pandoc`; PDF elevates only Poppler `pdftotext` through `--require-tool pdftotext` and extracts UTF-8 text with fixed layout-preserving arguments. All external converters use the same bounded no-shell path. The initialized repository remains `product_conversion_status: pending_review` and workflow resume returns a guarded `product-mark-ready` handoff; conversion never counts as semantic approval. PDF tables, diagrams, columns, and layout-dependent meaning must be compared against the archived source before closeout.

Use `--workflow-preset` for the standard fast path: `product-structure` expands through product structuring apply, `design-scaffold` expands through design scaffold apply, and `design-routing` adds read-only design authoring queue routing. A fresh `implementation-routing` run exposes readiness and design blockers but normally cannot reach implementation in the same process because Agent-authored design and authority reviews must happen first. After those queues are complete, rerun with `--resume --workflow-preset implementation-routing`; resume skips initialization, advances only when design is ready, and returns a read-only guarded-runner handoff. It never claims, executes, or closes an implementation task, and the CLI rejects combining this runner handoff with legacy implementation start/closeout flags. Presets never bypass normal dependency checks, write gates, skip behavior, or `workflow_preset_expanded_flags` audit evidence.

The script runs source-pack manifest verification, source-pack verification, `authority_skills.py --repair --check`, optional approved authority-skill apply, `env --repair --check`, `init --check`, write-mode initialization, and target-local verify/status/workflow-plan checks. `authority_skill_inventory.repair_plan` is always offline and no-write. `--approve-authority-installs` is ignored in `--check` mode; in write mode it installs only eligible locked missing skills before environment checks and target writes, then requires current provenance. `make test` uses `scripts/run_tests.py` to run each discovered `tests/test_*.py` module in an isolated local Python subprocess with a 15-minute timeout per module. Default concurrency is bounded by CPU count, Linux `MemAvailable` at 768 MiB per worker, and an eight-worker ceiling; environments without a readable memory signal retain the conservative four-worker ceiling. Use `python3 scripts/run_tests.py --workers <count>` only for an intentional local override. The runner emits stable module ordering, preserves complete failure output, rejects non-test module names, clears inherited GNU Make jobserver state, and disables bytecode writes. Use `make test-serial` to reproduce an ordering-sensitive failure with standard `unittest` discovery; it is a diagnostic fallback, not an additional release gate.

`bin/governance env --repair --check --json` previews environment repairs without writing `.governance/env-repair.md` or installing packages. It reports `would_repair`, system/package-manager/Git status, supported `install_commands`, structured `repair_commands` with `argv`, `cwd`, `writes_state`, and `approval_required`, ordered `repair_actions`, unsupported in-scope `manual_repairs`, `needs_escalation`, `repair_execution` with `status`, `can_auto_apply`, `install_attempted`, `install_failed`, `post_repair_missing_required`, `post_repair_missing_recommended`, and `next_step`, plus `repair_decision` with `decision`, `stop_before_workflow`, `runnable_action_ids`, `approval_action_ids`, and `manual_action_ids` for agent branching. Sort `repair_actions` by `sequence`; execute actions with `argv` only when `approval_required` is false or approval is explicit, and present `manual-repair` actions to the user. Run `bin/governance env --repair --json` only when the repair plan should be written or approved root package installation should proceed. Treat `applied_but_unresolved` as a stop state and inspect `post_repair_missing_required` before retrying package-manager repair. Missing required tools make `ok: false`; missing recommended tools make `ok: false` only with `--strict`. The repair command never calls `sudo`; supported apt installs run only when the process already has root privileges. ## Runtime Strategy

Core governance commands use POSIX shell wrappers and Python standard-library scripts so empty target folders can be initialized without package installation. Generated targets receive their own `bin/` and `scripts/` runtime plus `docs/agent-workflow/runtime-manifest.json`; after initialization, run checks from the target repository with `bin/governance verify .` or the target Makefile entries:

```bash
make verify-governance
make verify-check
make governance-status
make workflow-plan
make work-package
make workflow-resume
make product-plan
make design-plan
make implementation-plan
make implementation-run-check
make check-env
make repair-env-check
make project-env-plan
```

Generated targets also receive `docs/agent-workflow/workflow-pack/`, a manifest-verified snapshot of this pack's workflows, skills, references, and templates. Snapshot `VERSION`, both manifests' `pack_version`, and state `workflow_pack_version` must match. `verify` fails if the identity is invalid or drifted, or if a required runtime or workflow-pack snapshot file is missing, omitted from its manifest, or modified. From this source pack, run `bin/governance runtime refresh <target> --check --json` to inspect `would_refresh`, `would_remove`, the structured `version_transition`, and `migration_plan` without writing. The plan lists exact managed files, `preserved_project_document_roots`, ordered argv, and rollback requirements. Normal same-version, compatible-upgrade, and clean legacy-install transitions can then use `bin/governance runtime refresh <target> --json`. A breaking upgrade, rollback, version replacement, or conflicting/invalid installed-version evidence requires review of the check result and `bin/governance runtime refresh <target> --approve-version-transition --json`; without approval, write mode fails before changing target files. Refresh records the source version and replaces only generated `bin/`, `scripts/`, `docs/agent-workflow/runtime-manifest.json`, and workflow-pack snapshot files without rewriting product, design, planning, or implementation documents.

Node.js belongs in project-specific documentation and frontend tooling. Rust is reserved for optional stable accelerators after verification rules mature. See `references/runtime-strategy.md`.

## State File

Generated target repositories contain:

```text
.governance/state.json
```

The state file records the current workflow-pack version, workflow phase, project profile, product source, generated archive path, product import readiness, and last verification result. `bin/governance status <target>` prints the same key fields for quick human review.
